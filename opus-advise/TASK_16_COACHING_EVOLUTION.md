# TASK_16: AI コーチング進化 — 問いかけ型 + パイプライン管理

> **目的**: MEDDPICC コーチングを Salesforce 級の品質に引き上げ、SlackSFA の核心的差別化ポイントにする
> **優先度**: 🟡 高（SaaS 外販の武器）
> **依存**: TASK_15（応答速度改善）の後に実施
> **参照**: `/plan/slackSFA/MEDDICC.md`（Salesforce AE コーチング 5 カテゴリー）

---

## 1. 現状と課題

### 現状の実装（deal-coaching.service.ts）

- MEDDPICC 8要素を 0-100 でスコアリング ✅
- evidence（根拠）と status（done/partial/missing）を返す ✅
- suggested_actions で「具体的なアクション」を提案 ✅
- suggested_message で「顧客に送れるメッセージ例」を生成 ✅
- risk_factors でリスク要因を列挙 ✅
- deal_progress_history に履歴保存 ✅

### 足りていないもの（MEDDICC.md の 5 カテゴリーとの対比）

| Salesforce コーチング | 現状 | ギャップ |
|---------------------|------|---------|
| 1. 商談の進捗・戦略 | MEDDPICC スコア + アクション提案 | **問いかけ型**でない。答えを教えている |
| 2. パイプライン管理 | 日次ブリーフィングで件数・金額は出す | **3-4倍充足率**チェックがない。損切り提案もない |
| 3. 営業スキル | なし | **バリューセリング**の観点がない |
| 4. 社内リソース活用 | なし | 小規模チーム向けには不要（スコープ外） |
| 5. マインドセット | なし | AI の役割ではない（スコープ外） |

### 核心的な問題

> Salesforce のマネージャーは「答えを教える」のではなく「問いを立てる」

現在の AI は「決裁者との面談を設定してください」と**指示**している。
本来は「EB に会えない理由は何ですか？どうすれば会えると思いますか？」と**問いかける**べき。

## 2. 設計

### 2.1 問いかけ型コーチング

#### AI プロンプトの変更

```
変更前（suggested_actions）:
  "action": "決裁者との面談を設定してください"
  "reason": "予算権限者に直接提案することで受注確率が上がります"

変更後:
  "action_directive": "決裁者との接触を確認する"
  "coaching_question": "予算の最終決定権を持つ方にはお会いできていますか？もしまだなら、何が障壁になっていますか？"
  "if_yes": "素晴らしい。その方の判断基準は把握できていますか？"
  "if_no_hint": "チャンピオンに紹介を依頼するのが最短ルートです。次のメッセージ例を参考にしてください。"
  "suggested_message": "○○様、ご検討いただいているとのこと、ありがとうございます。最終的なご判断をされる方にもお話しさせていただければ..."
```

#### Slack 表示の変更

```
変更前:
  🔴 意思決定者（20/100）
  💡 決裁者との面談を設定してください
  理由: 予算権限者に直接提案することで受注確率が上がります

変更後:
  🔴 意思決定者（20/100）
  ❓ 予算の最終決定権を持つ方にはお会いできていますか？
     もしまだなら、何が障壁になっていますか？
  💡 ヒント: チャンピオンに紹介を依頼するのが最短ルートです
  📝 [この質問に回答する]  ← ボタン
```

#### 回答の記録

「この質問に回答する」ボタンを押すとモーダルが開き、回答を入力。
回答は `crm_notes` に保存され、次回のコーチングで AI がコンテキストとして参照する。

```
┌─────────────────────────────────────┐
│  コーチング回答                      │
│                                      │
│  質問:                               │
│  予算の最終決定権を持つ方には        │
│  お会いできていますか？              │
│                                      │
│  回答:                               │
│  ┌──────────────────────────────┐   │
│  │ まだ会えていません。担当の    │   │
│  │ 佐藤さん経由で紹介を依頼中   │   │
│  └──────────────────────────────┘   │
│                                      │
│            [キャンセル] [送信]       │
└─────────────────────────────────────┘
```

保存時:
1. `crm_notes` に `note_type = 'coaching_response'` で INSERT
2. `memo` に「Q: {質問}\nA: {回答}」を記録
3. センチメント分析 → 確率更新をトリガー
4. 回答内容に応じて AI が follow-up アクションを提示（次回コーチング時）

### 2.2 パイプライン充足率チェック

#### 概念

営業目標を達成するには、目標額の **3-4 倍** のパイプライン（open 商談の合計金額）が必要。

```
目標売上:    ¥10,000,000/月
必要パイプライン: ¥30,000,000〜¥40,000,000
現在の open 合計: ¥18,000,000
充足率:      45-60%  → ⚠️ 不足
```

#### DB: テナント設定に目標額を追加

```sql
-- 017_pipeline_target.sql
ALTER TABLE tenant_plans
  ADD COLUMN monthly_target_yen NUMERIC(12,0) DEFAULT 0,
  ADD COLUMN pipeline_ratio     NUMERIC(3,1) DEFAULT 3.0;  -- 目標の何倍必要か
```

#### 日次ブリーフィングに追加

```
既存:
  📊 本日のブリーフィング
  進行中: 8件 / ¥18,000,000
  今週クローズ予定: 2件

追加:
  📊 パイプライン充足率
  ████████░░░░░░░░░░░░ 45%
  目標 ¥10M × 3.0倍 = 必要 ¥30M / 現在 ¥18M
  ⚠️ パイプラインが不足しています。新規商談の創出を検討してください。
```

#### Web ダッシュボードに追加

Dashboard.vue に「パイプライン充足率」ゲージを追加。

### 2.3 損切り提案

#### ルール

| 条件 | 通知 |
|------|------|
| 30日以上未接触 + win_probability < 20% | 「この商談をクローズ・ロストにすべきか検討してください」 |
| 60日以上未接触（確率不問） | 「この商談は 60 日間動きがありません。継続する理由はありますか？」 |
| クローズ予定日を 30 日以上超過 | 「クローズ予定を過ぎています。日程を見直すか、損切りを検討してください」 |

#### 表示場所

- 日次ブリーフィングの末尾に「損切り候補」セクション
- AI Flow 詳細モーダルに「⚠️ 長期滞留」バッジ

#### Slack 通知

```
⚠️ 損切り検討が必要な商談

1. *ABC社 クラウド移行* — 45日未接触 / win: 15%
   ❓ この商談を継続する理由はありますか？
   [継続する（理由を記入）] [クローズ・ロスト]

2. *DEF社 基幹システム更新* — クローズ予定を32日超過
   ❓ 新しいクローズ予定日はいつですか？
   [日程更新] [クローズ・ロスト]
```

「継続する」→ モーダルで理由を入力 → メモに記録 + 時間減衰リセット
「クローズ・ロスト」→ `deals.stage = 'lost'` に更新 + 確認

### 2.4 競合管理（deals テーブル拡張）

```sql
-- 017_pipeline_target.sql（同じマイグレーション）
ALTER TABLE deals
  ADD COLUMN competitors JSONB DEFAULT '[]',       -- ["SalesForce", "HubSpot"]
  ADD COLUMN differentiators JSONB DEFAULT '[]';   -- ["価格", "AI分析"]
```

コーチング時に「競合は把握できていますか？」の問いに対して、回答を `competitors` に蓄積。
次回のコーチングで「前回 SalesForce が競合と伝えていましたが、差別化ポイントは明確ですか？」と問える。

## 3. AI プロンプト改修

### 3.1 コーチング評価プロンプト（deal-coaching.service.ts）

現在の `suggested_actions` を以下に拡張:

```json
{
  "overall_score": 65,
  "elements": { ... },
  "coaching_items": [
    {
      "priority": "high",
      "category": "economic_buyer",
      "category_label": "意思決定者",
      "coaching_question": "予算の最終決定権を持つ方にはお会いできていますか？",
      "if_answered_yes": "その方の判断基準は把握できていますか？",
      "if_answered_no": "チャンピオンに紹介を依頼するのが最短ルートです。",
      "action_hint": "次回のミーティングで紹介を依頼する",
      "suggested_message": "○○様、最終的なご判断をされる方にもお話しさせていただければ..."
    }
  ],
  "pipeline_note": "この商談が受注になっても目標の15%です。並行して新規開拓が必要です。",
  "stale_warning": null,
  "risk_factors": ["競合情報が不明", "クローズ予定まで残り12日"]
}
```

パイプライン充足率 + 損切り候補を追加:

## 7. 期待されるビジネスインパクト

| 観点 | 効果 |
|------|------|
| **営業の行動変容** | 指示ではなく問いかけにより、営業担当が自ら考える習慣がつく |
| **商談の精度向上** | パイプライン充足率の可視化で「量」の不足を早期発見 |
| **不良在庫の排除** | 損切り提案で滞留商談を整理し、リソースを有望案件に集中 |
| **SaaS 差別化** | 「AI が営業マネージャーの代わりに問いかける」は他社 CRM にない価値 |
| **データ蓄積** | コーチング回答が crm_notes に蓄積 → AI の精度が使うほど上がる |
